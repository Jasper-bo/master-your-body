## 2. 产品概述

### 2.1 背景

随着健康意识的普及，越来越多的用户希望通过数字化手段管理日常饮食、运动和身体状况。然而，市面上大多数健康管理工具存在以下痛点：
- **记录门槛高**：手动输入食物和训练数据流程繁琐，导致用户难以坚持。
- **数据孤岛**：饮食、运动、身体指标分散在不同应用，缺乏统一视图。
- **反馈滞后**：用户无法直观看到每日努力与健康成果的关联，缺乏持续动力。
- **模板化严重**：界面千篇一律，缺乏个性化和情感化设计。

VitalPulse 旨在打造一款**一体化、智能化、高颜值**的健康管理工具，通过 AI 拍照识别、自动化数据聚合、可视化的健康评分体系，降低记录成本，提升用户黏性。

### 2.2 产品目标

1. **降低记录成本**：AI 拍照识别食物 + 一键添加常用动作，让记录耗时控制在 30 秒以内。
2. **数据可视化**：将复杂的营养、训练、身体指标转化为直观的进度条、环形图、折线图。
3. **游戏化激励**：通过每日健康评分与等级体系，让用户对每日健康行为产生正向反馈。
4. **个性化指导**：基于用户身体数据和目标，自动计算每日营养与训练建议。

### 2.3 用户画像

| 用户类型 | 特征 | 核心诉求 |
|----------|------|----------|
| **健身入门者** | 20-30 岁，刚开始关注饮食和训练，对宏量营养素概念模糊 | 简单易懂的记录方式，自动计算目标，避免复杂操作 |
| **自律健身者** | 25-35 岁，有规律的训练习惯，关注身体成分变化 | 精确的营养追踪、训练容量记录、历史趋势分析 |
| **减脂人群** | 25-40 岁，以体重管理为主要目标 | 热量控制、便捷的食物记录、每日打卡提醒 |
| **健康维护者** | 30-45 岁，工作繁忙，希望维持基本健康水平 | 快速记录、直观的健康评分、低使用门槛 |

## 3. 产品目标与成功指标

### 3.1 北极星指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 日活跃记录率（DARR） | ≥ 40% | 每日至少完成 1 次饮食或训练记录的活跃用户占比 |
| 7 日留存率 | ≥ 35% | 新用户注册后第 7 天仍活跃的比例 |
| 单次记录完成时间 | ≤ 30 秒 | 从打开应用到完成一次食物或训练记录的平均耗时 |

### 3.2 功能指标

| 功能 | 指标 | 目标值 |
|------|------|--------|
| AI 拍照识别 | 识别准确率（Top-1） | ≥ 75% |
| AI 拍照识别 | 平均响应时间 | ≤ 3 秒 |
| 健康评分 | 日评分计算覆盖率 | 100%（有数据的用户） |
| 仪表盘加载 | 首屏加载时间 | ≤ 1.5 秒 |

### 5.2 饮食记录子流程

```
[饮食与营养页面]
    │
    ├──→ [AI 拍照]
    │       │
    │       ├──→ 前端选择 JPG/PNG 文件（≤10MB）
    │       ├──→ 前端 multipart 上传至本地后端 API
    │       ├──→ 后端接收图片，进行类型/大小校验
    │       ├──→ 后端将图片 Base64 编码，调用 DeepSeek API（Vision 模型）
    │       ├──→ DeepSeek 返回：食物名、蛋白质(g)、碳水(g)、脂肪(g)
    │       ├──→ 后端解析响应，热量由前端计算（蛋白质×4 + 碳水×4 + 脂肪×9）
    │       ├──→ 后端不保存图片，仅返回识别结果
    │       ├──→ 前端展示结果卡片，用户可修改数据
    │       └──→ 用户确认后进入餐食记录流程
    │
    └──→ [手动录入]
            │
            ├──→ 选择分类（主食 / 肉类 / 蔬菜 / 油脂）
            ├──→ 选择具体食物，输入克数
            ├──→ 油脂用量 4 档单选（无油/少油/中等/多油）
            ├──→ 前端实时计算热量预览
            └──→ 确认记录时选择餐别（智能推断默认值）
```

## 6. 功能需求详述

### 6.1 全局功能

#### 6.1.1 认证体系（注册 / 登录）

**功能描述：**
- 支持手机号注册与登录，纯数字 8-15 位。
- 密码至少 6 位，使用 bcrypt 哈希存储。
- 图形验证码 4 位数字，防止暴力破解。
- 未登录用户强制跳转至 `/login`。

**交互逻辑：**
- `/register`：输入手机号、密码、确认密码、图形验证码 → 前端校验 → 调用注册 API → 成功直接进入个性化计划弹窗。
- `/login`：输入手机号、密码、图形验证码 → 前端校验 → 调用登录 API → 成功返回 JWT → 跳转 `/dashboard`。
- JWT Access Token 有效期 24 小时，Refresh Token 有效期 7 天。
- Token 过期后自动使用 Refresh Token 续期，续期失败则跳转登录页。

**边界情况：**
- 手机号已注册：提示"该手机号已注册，请直接登录"。
- 密码错误 5 次：临时锁定账号 15 分钟。
- 图形验证码过期（5 分钟）：提示"验证码已过期，请重新获取"。

#### 6.1.2 顶部导航栏

**功能描述：**
- 固定于页面顶部的导航栏，包含品牌标识、页面入口。

**交互逻辑：**
- 当前所在页面，导航项呈现高亮状态（下划线或背景色变化）。
- 点击导航项平滑切换至对应页面，使用前端路由（无刷新）。
- 导航入口：`/dashboard`（仪表盘）、`/nutrition`（饮食）、`/training`（训练）、`/settings`（设置）。
- 未登录用户：仅显示"登录 / 注册"按钮，点击跳转对应页面。

**边界情况：**
- 网络异常时点击导航：提示"网络连接异常，请检查网络后重试"。

#### 6.1.3 个性化计划弹窗（首次引导）

**功能描述：**
- 新用户注册成功后**立即弹出，不可关闭/跳过，必须填完所有字段才能进入系统**。
- 老用户可通过仪表盘"更新计划"浮动按钮重新打开，修改身体数据。
- 后端根据提交数据自动计算 BMI、BMR（Mifflin-St Jeor 公式）、TDEE、每日热量及三大营养素目标。
- 饮水目标默认 2000ml，睡眠目标默认 7.5h。

**字段与校验：**

| 字段 | 类型 | 校验规则 | 必填 |
|------|------|----------|------|
| 身高 | 数字输入 | 50-300 cm | 是 |
| 体重 | 数字输入 | 20-300 kg | 是 |
| 年龄 | 数字输入 | 10-100 岁 | 是 |
| 性别 | 单选 | 男 / 女 / 其他 | 是 |
| 运动频率 | 单选 | 久坐不动 / 轻度 1-2 天 / 中度 3-5 天 / 极度 6-7 天 | 是 |
| 健身目标 | 单选 | 减脂 / 维持 / 增肌 | 是 |

**交互逻辑：**
- 弹窗以模态遮罩形式居中展示，背景模糊（backdrop-filter: blur）。
- **不显示关闭按钮（×），不响应 ESC 键，点击遮罩不关闭。**
- 填写完成后点击"生成我的计划"，前端先做字段校验，通过后调用后端 API。
- 后端计算并写入 `user_profiles` 表。
- 成功后在弹窗内展示生成的计划摘要（每日热量目标、三大营养素分配、饮水目标、睡眠目标），用户点击"进入首页"关闭弹窗，跳转 `/dashboard`。

**边界情况：**
- 用户中途关闭浏览器：下次进入重新弹出，已填写字段不保留（因未登录完成前无用户身份）。
- API 计算失败：展示错误提示，允许重新提交。
- 老用户点击"更新计划"：弹窗预填充当前数据，提交后更新 `user_profiles` 并重新计算未来日期的目标值。

### 6.2 主页 / 仪表盘（`/dashboard`）

#### 6.2.1 日期与欢迎语

**功能描述：**
- 展示当前日期（长格式，如"2026年4月28日 星期二"）和个性化欢迎语。

**交互逻辑：**
- 欢迎语根据时间段变化：
  - 05:00-11:59："早上好，{昵称}，开启充满活力的一天！"
  - 12:00-17:59："下午好，{昵称}，保持状态！"
  - 18:00-21:59："晚上好，{昵称}，今天训练了吗？"
  - 22:00-04:59："夜深了，{昵称}，注意休息。"
- 日期支持左右箭头切换前后日期，切换后全页数据联动刷新。

**边界情况：**
- 切换到未来日期：允许查看，但所有打卡项置为不可编辑，提示"未来日期无法记录"。
- 切换到过去无数据日期：展示空状态"当日无记录"，并提供"补录"入口跳转至对应页面。

#### 6.2.2 每日营养卡片

**功能描述：**
- 卡片展示蛋白质、碳水、脂肪的当前摄入值与目标值，以进度条形式呈现。

**展示规则：**

| 营养素 | 目标来源 | 当前值来源 | 进度条颜色 |
|--------|----------|------------|------------|
| 蛋白质 | `user_profiles.daily_protein_target_g` | `daily_nutrition.total_protein_g` | 蓝色 |
| 碳水 | `user_profiles.daily_carbs_target_g` | `daily_nutrition.total_carbs_g` | 橙色 |
| 脂肪 | `user_profiles.daily_fat_target_g` | `daily_nutrition.total_fat_g` | 黄色 |

**交互逻辑：**
- 进度条实时计算百分比：当前值 / 目标值 × 100%。
- 超过 100% 时进度条满格，数值变为红色警示色，提示"已超标 {超出值}g"。
- 点击卡片任意区域跳转至"/nutrition"页面。

**边界情况：**
- 用户未设置目标（跳过了个性化计划）：进度条显示"--"，目标显示"未设置"，点击引导至个性化弹窗。
- 当日无任何餐食记录：当前值显示 0，进度条为 0%。

#### 6.2.3 身体指标速览

**功能描述：**
- 横向排列的三个指标卡片：BMI、BMR、今日完成率。

**计算规则：**
- **BMI**：基于 `user_profiles` 最新身高体重实时计算，保留 1 位小数。
- **BMR**：基于 Mifflin-St Jeor 公式实时计算，向下取整。
- **今日完成率**：`health_checklist` 中 3 项打卡的完成百分比（每项约 33.3%）。

**交互逻辑：**
- BMI 数值旁显示分类标签（偏瘦/正常/超重/肥胖），颜色区分（绿/橙/红）。
- 点击 BMI 卡片：展开简要说明"BMI 是什么？"浮层。
- 点击 BMR 卡片：提示"基础代谢率是指人体在静息状态下维持生命所需的最低热量"。

**边界情况：**
- 用户未填写身高/体重：显示"--"，点击引导至个性化计划弹窗。

#### 6.2.4 健康评分清单

**功能描述：**
- 垂直列表展示 3 项每日健康打卡，每项包含图标、名称、完成状态、快捷操作。

**打卡项定义：**

| 项 | 图标 | 完成条件 | 快捷操作 |
|----|------|----------|----------|
| 水分补充 | 水滴 | `health_checklist.water_intake >= 目标值` | 点击弹出饮水量输入（+250ml / +500ml / 自定义，单位 ml） |
| 睡眠质量 | 月亮 | `health_checklist.sleep_duration >= 目标值` | 点击弹出睡眠时长输入（单位 h，默认 7.5h） |
| 运动锻炼 | 跑步 | `health_checklist.exercise_done = true` | 点击跳转训练页面；训练完成自动勾选 |

**交互逻辑：**
- 未完成项左侧显示空心圆圈，已完成显示绿色实心勾选。
- 点击项右侧可展开快捷操作面板。
- 标记完成后触发健康评分重新计算，仪表盘总分实时更新。

**边界情况：**
- 用户标记睡眠完成但未输入时长：提示"请输入睡眠时长以精准计算评分"。
- 用户当日已打卡但切换至过去日期：只读展示，不可修改。

#### 6.2.5 总评分环形图

**功能描述：**
- 中央展示当日健康总评分（0-100），外圈为环形进度图，评分下方显示等级（A/B/C/D）。

**等级映射：**

| 分数 | 等级 | 颜色 |
|------|------|------|
| ≥ 90 | A | 青绿色 |
| 75-89 | B | 蓝色 |
| 60-74 | C | 橙色 |
| < 60 | D | 红色 |

**交互逻辑：**
- 环形图使用 SVG 或 Canvas 绘制，动画时长 1 秒（ease-out）。
- 点击环形图：展开评分详情面板，展示 3 个分项得分（营养 / 运动 / 睡眠水分）及打卡加成。
- 动画仅在首次进入视口时触发一次（IntersectionObserver），避免重复播放。

**边界情况：**
- 当日无任何数据：显示"--"，环形图为灰色空心，提示"开始记录以获取评分"。
- 评分计算中：展示骨架屏（shimmer 效果）。

### 6.3 饮食与营养页面（`/nutrition`）

#### 6.3.1 AI 拍照识别食物

**功能描述：**
- 用户上传食物照片，本地后端调用 DeepSeek API 进行识别，返回食物名及三大营养素含量，前端计算热量。图片不保存，结果可修改。

**完整调用链路：**

```
[用户点击"拍照识别"]
    │
    ▼
[前端] 唤起系统文件选择器，限制文件类型 .jpg / .jpeg / .png
    │
    ▼
[前端] 校验文件大小 ≤ 10MB，图片分辨率 ≥ 300×300 px
    │
    ▼
[前端] 将图片封装为 multipart/form-data，上传至本地后端 API
       POST /api/ai/recognize-food
    │
    ▼
[后端] 接收请求，进行服务端二次校验：
       - MIME 类型校验（image/jpeg、image/png）
       - 文件大小校验（≤ 10MB）
       - 图片内容基础校验（非空、可读）
    │
    ▼
[后端] 将图片 Base64 编码，构建 DeepSeek API 请求：
       - 模型：deepseek-vision 或等效多模态模型
       - 消息内容：
         {
           "role": "user",
           "content": [
             { "type": "text", "text": "识别这张图片中的食物，返回 JSON 格式：{food_name: string, protein_g: number, carbs_g: number, fat_g: number}。只返回纯 JSON，不要其他文字。" },
             { "type": "image_url", "image_url": { "url": "data:image/jpeg;base64,{base64}" } }
           ]
         }
    │
    ▼
[DeepSeek API] 返回识别结果（JSON 格式）
    │
    ▼
[后端] 解析响应，提取 food_name、protein_g、carbs_g、fat_g
       - 对数值进行合理性校验（非负、有上限）
       - 若解析失败，返回错误码
    │
    ▼
[后端] 不保存图片文件，不写入图片存储
       仅将识别结果返回给前端
    │
    ▼
[前端] 接收结果，展示识别结果卡片：
       - 食物名称
       - 蛋白质 / 碳水 / 脂肪（可编辑）
       - 热量由前端实时计算：calories = protein×4 + carbs×4 + fat×9
       - 置信度标签（基于 DeepSeek 响应中的确定性描述）
    │
    ▼
[用户] 可执行以下操作：
       - 确认正确：将数据带入餐食记录流程
       - 手动修正：修改食物名称或各项营养数值
       - 重新拍摄：清空当前结果，重新上传
```

**边界情况：**
- AI 识别失败（网络超时 / DeepSeek 返回异常）：展示"识别失败"状态，提供"重试"和"手动录入"两个选项。
- AI 识别出多种食物：展示为多卡片列表，用户可分别确认/删除每一项。
- 用户连续快速上传：限制同时只能有 1 个识别任务在进行中，后续任务排队或提示"请等待当前识别完成"。
- DeepSeek 返回非 JSON 格式：后端尝试正则提取，失败则返回"识别结果异常，请重试或手动录入"。

#### 6.3.3 手动食物录入

**功能描述：**

**分类结构：**

**A. 主食类**

**C. 蔬菜类**

**D. 油脂用量**

**边界情况：**

- 用户输入极大值（如 99999g）：前端限制最大 5000g，超出则自动截断并提示"单次记录上限 5000g"。
- 用户需要的食物不在列表中：提示"后续版本支持食物搜索"。

**功能描述：**

**交互逻辑：**

  - 展示食物清单（名称 + 克数 + 热量）

## 7. 非功能需求

### 7.1 性能需求

| 场景 | 指标 | 目标值 |
|------|------|--------|
| 仪表盘首屏加载 | FCP | ≤ 1.5s |
| 仪表盘首屏加载 | LCP | ≤ 2.5s |
| 页面交互响应 | INP | ≤ 200ms |
| 切换日期 | 数据刷新 | ≤ 500ms |
| AI 拍照识别 | 端到端耗时 | ≤ 5s（P95） |

**优化措施：**
- 仪表盘数据采用服务端聚合（`daily_nutrition`、`health_scores`），避免实时 JOIN。
- 图片资源使用 WebP/AVIF 格式，懒加载非首屏图片。
- Next.js 服务端组件（Server Components）减少客户端 JavaScript 体积。
- AI 识别使用本地后端中转，避免前端直接暴露 DeepSeek API Key。

### 7.2 安全需求

| 类别 | 要求 |
|------|------|
| 认证 | JWT（Access Token + Refresh Token），Access Token 有效期 24 小时，Refresh Token 有效期 7 天 |
| 密码 | bcrypt 哈希，cost factor ≥ 12 |
| 传输 | 全站 HTTPS，HSTS 启用 |
| 文件上传 | 限制类型（jpg/png），限制大小（≤10MB），服务端二次校验 MIME 类型 |
| AI 图片 | 对用户上传图片进行类型和大小校验，不保存原始图片，仅传输至 DeepSeek |
| 隐私数据 | 用户身体数据、饮食记录、训练记录属于敏感数据，API 响应需严格鉴权，禁止越权访问 |
| 输入防注入 | 所有用户输入使用参数化查询（Prisma ORM），禁止 SQL 拼接；前端输入做长度和类型限制 |
| CSP | 配置 Content-Security-Policy，禁止内联脚本，限制图片/字体来源 |

### 7.3 兼容性需求

| 平台 | 要求 |
|------|------|
| 桌面 Chrome | 支持最近 2 个主版本 |
| 桌面 Safari | 支持最近 2 个主版本 |
| 桌面 Firefox | 支持最近 2 个主版本 |

**布局策略：**
- 桌面端优先，内容区居中最大 1200px。
- 不支持移动端适配（v1.0 范围外）。

### 7.4 可用性需求

- **无障碍（A11y）**：
  - 所有图片含 `alt` 文本。
  - 交互元素支持键盘操作（Tab 导航、Enter/Space 触发）。
  - 颜色对比度符合 WCAG AA 标准（≥ 4.5:1）。

### 8.1 关键数据流

```
[用户录入食物]
    │
    ├── 写入 meal_records（餐食级）
    ├── 写入 meal_foods（食物级，含计算后营养）
    ├── 触发聚合 → 更新 daily_nutrition
    └── 触发评分 → 更新 health_scores

[用户提交训练]
    │
    ├── 写入 workout_records（训练级）
    ├── 写入 workout_exercises（组数级）
    ├── 触发更新 → health_checklist.exercise_done = true
    └── 触发评分 → 更新 health_scores

[用户修改身体数据]
    │
    ├── 更新 user_profiles
    ├── 重新计算 daily_*_target 字段
    ├── 更新饮水目标（默认 2000ml）和睡眠目标（默认 7.5h）
    └── 级联更新所有未来日期的 daily_nutrition 目标值（历史保持不变）
```

## 9. 风险与依赖

### 9.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI 食物识别准确率不足 | 用户信任度下降，记录体验差 | 引入置信度提示，低置信结果强制用户确认；DeepSeek 返回结果可由用户手动修正；建立常见食物本地校验规则 |
| 营养数据库覆盖不全 | 用户找不到常见食物，手动录入受阻 | 优先覆盖高频食物（中餐 Top 200）；v1.1 支持食物搜索功能；v1.1 后支持用户自定义食物 |
| 高并发下聚合表更新延迟 | 仪表盘数据不一致 | 使用数据库触发器保证最终一致性；聚合表更新走异步处理；前端展示时显示"数据更新中"状态 |
| DeepSeek API 不可用或限流 | AI 拍照功能完全不可用 | 降级为纯手动录入；本地缓存高频食物识别结果；实现指数退避重试机制 |

### 9.2 业务风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 用户留存率低 | 产品价值未达预期 | 强化健康评分的游戏化反馈；仪表盘数据可视化增强成就感；训练回顾提供正向激励 |
| 健康评分算法被质疑不科学 | 用户不信任 | 算法透明化（详情页展示分项计算过程）；提供免责声明"仅供参考，不构成医疗建议" |
| 用户隐私顾虑 | 不愿上传身体数据 | 纯本地部署，所有敏感数据存储于本地 SQLite 文件；明确的隐私政策；用户完全掌控数据 |

### 9.3 外部依赖

| 依赖 | 用途 | 备选方案 |
|------|------|----------|
| DeepSeek API（Vision 模型） | 食物拍照识别 | 降级为手动搜索；接入其他国产多模态模型（如通义千问、智谱 GLM-4V） |
| Next.js 框架 | 全栈开发基础 | 锁定版本，避免 major 升级带来的 Breaking Change |
| Prisma ORM | 数据库访问 | 原生 SQLite 驱动（迁移成本高，作为最后手段） |

## 11. 附录

### 11.1 术语表

| 术语 | 解释 |
|------|------|
| BMI | Body Mass Index，身体质量指数 |
| BMR | Basal Metabolic Rate，基础代谢率 |
| TDEE | Total Daily Energy Expenditure，每日总能量消耗 |
| 宏量营养素 | 蛋白质、碳水化合物、脂肪的统称 |
| Bento 网格 | 一种不规则的卡片网格布局，大小不一 |
| JWT | JSON Web Token，用于认证的令牌格式 |
| Mifflin-St Jeor | 计算 BMR 的标准公式，比 Harris-Benedict 更准确 |

### 11.2 参考算法

所有核心计算逻辑（BMI、BMR、TDEE、营养目标分配、健康评分算法）的伪代码及详细说明，详见 `data-schema.md` 第 4 节。

**核心算法速查：**

- **BMI** = 体重(kg) / (身高(m))²
- **BMR（Mifflin-St Jeor）**：
  - 男：10 × 体重(kg) + 6.25 × 身高(cm) - 5 × 年龄 + 5
  - 女：10 × 体重(kg) + 6.25 × 身高(cm) - 5 × 年龄 - 161
- **TDEE** = BMR × 活动系数（久坐 1.2 / 轻度 1.375 / 中度 1.55 / 极度 1.725）
- **热量目标**：
  - 减脂：TDEE - 300~500 kcal
  - 维持：TDEE
  - 增肌：TDEE + 300~500 kcal
- **三大营养素分配**（按健身目标）：
  - 减脂：蛋白质 40% / 碳水 30% / 脂肪 30%
  - 维持：蛋白质 30% / 碳水 40% / 脂肪 30%
  - 增肌：蛋白质 30% / 碳水 50% / 脂肪 20%
