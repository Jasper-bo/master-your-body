VitalPulse 健康管理系统 — 软件工程课程设计答辩

一、项目概述

VitalPulse 是一个个人健康管理 Web 应用，帮助用户记录每日饮食、训练、睡眠和饮水数据，通过健康评分系统给出综合反馈。项目采用前后端一体化架构，基于 Next.js 15 App Router 构建，已部署到 Vercel，数据库使用 Supabase 托管的 PostgreSQL，代码托管在 GitHub。

核心功能模块包括四个主要板块：仪表盘（Dashboard）汇总当日所有健康数据、饮食记录（Nutrition）录入餐食并自动计算营养、训练记录（Training）记录力量或有氧训练、个人设置（Settings）管理身体档案。外加一套完整的手机号注册/登录认证体系。

项目技术选型如下：

层次技术栈说明
前端框架React 19 + Next.js 15 (App Router)基于文件系统路由，支持 RSC

二、前端设计

2.1 页面路由结构

使用 Next.js App Router 的文件系统路由，通过路由组实现不同布局：

路由组页面路径布局类型说明
(auth) — 未登录用户/login居中卡片式布局登录页
(auth)/register居中卡片式布局注册页（含档案弹窗）
(main) — 已登录用户/dashboard顶部导航 + 内容容器健康仪表盘
(main)/nutrition顶部导航 + 内容容器饮食记录
(main)/training顶部导航 + 内容容器训练记录
(main)/settings顶部导航 + 内容容器个人设置

/ 根路径自动重定向到 /dashboard。

路由保护通过 Next.js 中间件（middleware.ts）实现：已登录用户访问 /login 或 /register 时自动重定向到仪表盘；未登录用户访问受保护页面时，若有 refresh token 则尝试无感刷新，否则重定向到登录页并附带 redirect 参数以便登录后回跳。

2.3 核心交互流程

  1. 注册流程：填写手机号 + 密码 + 确认密码 → 调用 POST /api/auth/register → 注册成功 → 强制弹出 ProfileModal 档案弹窗（高度、体重、年龄、性别、活动水平、健身目标）→ 提交 PUT /api/users/me/plan → 后端根据公式计算个性化营养目标（BMR × 活动系数 = TDEE，再按增肌/减脂/维持调整热量及三大宏量素配比）→ 跳转仪表盘。
  2. 登录流程：填写手机号 + 密码 → POST /api/auth/login → 服务端验证后通过 Set-Cookie 下发 JWT 双 token → 前端清理旧状态后跳转。安全机制：连续 5 次密码错误锁定账号 15 分钟。
  3. 仪表盘：GET /api/dashboard → 一次性返回当日健康全貌（营养摄入 vs 目标、身体指标 BMI/BMR/TDEE、健康评分四维小分 + A-D 等级、7 日营养趋势图数据、三项打卡状态），前端渲染为卡片 + 进度条 + 折线图 + 勾选框。
  4. 饮食记录：选择餐型（早/中/晚/加餐）→ 按分类选食物 → 填克数（主食 10-2000g，肉类/蔬菜 10-1000g）→ 选油脂量（无油/少油/中等/多油）→ POST /api/nutrition/meals → 后端在事务中创建 MealRecord + MealFood 条目 → 自动重算当日 DailyNutrition。
  5. 训练记录：选部位 → 选动作 → 逐组填重量(0-500kg)和次数(1-100)或时长(1-600min) → POST /api/training/quick-log → 后端自动推算出总时长、总训练量（volume = 重量×次数的累计）→ 在事务中创建 WorkoutRecord + WorkoutExercise 条目 → 自动更新 HealthChecklist.exerciseDone。

三、后端设计

3.1 架构概述

后端基于 Next.js API Routes，共 23 个路由文件、25 个 API 端点。采用服务端函数式架构：API 路由层负责请求校验和响应序列化，核心业务逻辑集中在 src/lib/server/ 下的领域模块中。所有 API 返回统一的 JSON 信封格式：

{
  success: boolean,    // 请求是否成功
  data?: T,            // 业务数据（成功时）
  error?: string,      // 错误信息（失败时）
  meta?: {             // 分页元数据（列表接口）
    total: number,
    page: number,
    limit: number,
    totalPages: number,
    hasNext: boolean,
    hasPrev: boolean
  }
}

3.4 健康评分算法（核心业务逻辑）

健康评分系统位于 dashboard.ts 中的 calculateHealthScore 函数，采用 4 维度 × 25 分 = 100 分制，最终映射为 A / B / C / D 等级：

维度分值范围计算方式目标来源
营养0-25热量达标分 (60%权重) + 三大宏量素均衡分 (40%权重)，热量偏差超 10% 即清零用户档案个性化计算
运动0-25训练时长 ≥ 30 分钟 → 25 分，否则按比例折算固定 30 分钟标准
睡眠0-25实际睡眠 / 目标睡眠 × 25，上限 25（允许超额）用户档案 sleepTargetHours
水分0-25实际饮水 / 目标饮水 × 25，上限 25用户档案 waterTargetMl

总分 = 营养分 + 运动分 + 睡眠分 + 水分分。等级映射：≥90 → A，≥75 → B，≥60 → C，<60 → D。

评分结果通过 prisma.healthScore.upsert 持久化到 health_scores 表，供后续查询和趋势分析。

3.5 认证安全机制

安全措施实现方式
密码保护bcrypt 12 轮哈希，不存储明文
JWT 签名HMAC-SHA-256，access token (24h) 和 refresh token (7d) 使用不同密钥签名
防暴力破解连续输错 5 次密码 → lockedUntil = now + 15min，锁定期间拒绝登录
Refresh Token Rotation每次使用 refresh token 时吊销旧 token、签发新 token pair，防重放攻击
Token 安全存储Refresh token 数据库中仅存 SHA256 哈希值，即使数据库泄露也无法伪造
Cookie 安全httpOnly (JS 不可读，防 XSS) + SameSite=Lax (防 CSRF) + Secure (HTTPS only, 生产环境)
输入校验所有 API 入参经过 schema 校验（类型、范围、格式），非法输入在到达数据库前被拦截
数据一致性饮食和训练的创建/删除操作均在 Prisma $transaction 中完成，多表更新原子化

四、数据库设计

采用 PostgreSQL + Prisma ORM，共 17 张表 + 6 个枚举类型。数据库托管在 Supabase，通过 PgBouncer 连接池连接（端口 6543），Prisma schema 迁移走直连（端口 5432）。

4.4 数据库设计亮点

  • 一人一天一行：DailyNutrition、HealthChecklist、HealthScore 三张表均使用 (userId, recordDate) 唯一约束，通过 upsert 更新，保证数据不会重复
  • 营养冗余计算：MealFood 表中存储 calculatedCalories/Protein/Carbs/Fat，避免每次查询时根据克数和食物营养值重复计算
  • 食物/动作库可扩展：isSystem 区分系统预设和用户自定义，isActive 支持软删除，种子数据在应用启动时自动初始化
  • 级联删除：User 删除时 9 张关联子表通过 ON DELETE CASCADE 自动级联清理，保证数据一致性
  • 索引覆盖：所有高频查询字段（userId+date、categoryId、sortOrder、tokenHash、expiresAt、status）均已建立联合索引或单列索引

JWT_ACCESS_EXPIRYAccess Token 有效期24h
JWT_REFRESH_EXPIRYRefresh Token 有效期7d
BCRYPT_ROUNDS密码哈希强度12
LOGIN_MAX_FAILURES触发锁定的失败次数5
LOGIN_LOCKOUT_MINUTES锁定持续时间15
